# SRBI Extraction Pipeline — Full Project Plan for Codex

## What You Are Building

A production-grade AI pipeline that takes a folder of raw scraped text/HTML files about a company, extracts structured intelligence (company overview, funding, capacity gaps, pain points, recent trigger events, offering fit), tags every claim as PRIMARY / SECONDARY / INFERRED with source attribution, and outputs a single validated JSON blob that a backend team renders into an SRBI-style intelligence report.

The output JSON must mirror exactly what is shown in the sample SRBI reports: not a simple key-value dump, but a deeply sourced, confidence-scored intelligence document where every claim traces back to a specific URL with its source tier.

---

## Project Goal in One Sentence

Given a folder of scraped files about **one specific company** (the pipeline processes exactly one company per run), produce a JSON document that answers: what does this company do, what is broken in their operations, why are they under pressure right now, and why would they buy our product — all backed by cited, tiered sources.

**Important runtime constraint**: The pipeline is designed for one company at a time. The team will invoke it once per company, providing a single `company_id` and an `input_dir` containing only that company's scraped files. There is no batch mode, no queue, no parallel processing. This keeps the implementation simple, the FAISS index small and focused, and the cost per run predictable. If batch processing is ever needed, it is the caller's responsibility to loop over `run.py` externally.

## Technology Decisions — Final Choices and Why

### LangGraph over LangChain
Use **LangGraph** (`langgraph>=0.2`). The pipeline has distinct sequential stages with a shared state object that all nodes read from and write to. LangGraph's `StateGraph` with typed state is the correct abstraction for this. LangChain's LCEL chains are better for simple prompt-in-prompt-out flows; this pipeline is a stateful multi-stage process. LangGraph compiles cleanly to a directed graph, gives you free retry logic via conditional edges, and the state is a plain Python TypedDict that is easy to inspect and debug.

### FAISS over Qdrant
Use **FAISS** (`faiss-cpu`). The pipeline runs one company at a time, data volume is in the hundreds of documents not millions, and the team does not want infrastructure overhead. FAISS is a local in-memory index. Persist it per company to disk with `faiss.write_index`. Do NOT use Qdrant — it requires a running server process and adds operational complexity that is not justified here.

### Instructor over raw LLM calls
Use **Instructor** (`instructor>=1.0`) wrapping the Anthropic client. Instructor enforces Pydantic schema validation on every LLM response and auto-retries with the validation error fed back to the model when the output is malformed. This is what guarantees null-on-absence behavior: if the schema has `Optional[str]` and the evidence is missing, the model returns `None` and Instructor validates that correctly instead of letting the model hallucinate a value.

### Claude Sonnet as the extraction LLM
Use `claude-sonnet-4-20250514` via the Anthropic SDK. It is the best instruction-following model for structured extraction tasks. All Instructor calls go through this model.

### OpenAI text-embedding-3-small for embeddings
Use `text-embedding-3-small` via the OpenAI SDK. 1536 dimensions, cheap, fast, high quality. If the team wants fully local/free embeddings, swap to `sentence-transformers` with `all-MiniLM-L6-v2` — the FAISS dimension just changes from 1536 to 384 and nothing else in the pipeline changes. Keep this swap documented as a comment in `retrieval.py`.

### Pydantic v2 throughout
All models use Pydantic v2 syntax (`model_validator`, `field_validator`, `model_config`). No v1 compatibility shims.

## Product Knowledge Store — `products/` Directory

This is the missing piece that makes the Offering Fit section genuinely useful instead of generic. The LLM cannot produce a specific, accurate "what to supply" answer for a company unless it knows exactly what your products do, who they are for, and what problems they solve.

### How it works

The `products/` directory contains one JSON file per product your company sells. These files are NOT scraped data — they are maintained by your team and committed to the repo. They are the authoritative product context injected into the `OfferingFitCluster` extraction.

### Product Profile JSON Format

Each file in `products/` must follow this schema:

```json
{
  "product_id": "gridzy-app-store",
  "product_name": "Gridzy App Store",
  "one_liner": "Pre-built no-code operational apps for industrial enterprises — deploy without custom dev",
  "what_it_does": "A curated marketplace of production-ready apps (Tender & Bidding, Visitor Management, Task Management, Quality Workflows) that enterprises can deploy on-premise or on cloud without writing code.",
  "ideal_customer_profile": {
    "industries": ["manufacturing", "automotive", "industrial"],
    "company_size": "500+ employees",
    "pain_signals": [
      "IT backlog exceeding dev capacity",
      "Multiple plants with disconnected workflows",
      "No existing no-code platform",
      "CTO or VP Ops driving digital transformation mandate"
    ],
    "deployment_preference": "on-premise or hybrid"
  },
  "key_differentiators": [
    "On-premise deployment — production data never leaves company infrastructure",
    "Pre-built apps — no custom development required",
    "India-native pricing — not priced for Western enterprise budgets"
  ],
  "not_a_fit_if": [
    "Company already uses OutSystems or Mendix",
    "Pure cloud SaaS mandate with no on-prem option needed",
    "Single-location company with under 100 employees"
  ],
  "pricing_model": "Annual license per deployment, not per user"
}
```

Create one `.json` file per product. Names can be anything (`gridzy-app-store.json`, `gritwiz-decision-intel.json`). The pipeline loads ALL files in `products/` at startup.

### How Product Profiles Are Injected

In `node_extract`, when running the `OfferingFitCluster` extraction specifically, the pipeline:

1. Loads all product profiles from the `products/` directory at the start of `node_extract` (not per-call — load once, reuse)
2. Formats them as a structured product context block
3. Appends this block to the context alongside the retrieved company chunks

The prompt for `OfferingFitCluster` explicitly tells the model:

```
You have two types of context below:
A) COMPANY SIGNALS — scraped evidence about this company's pain points, gaps, and triggers
B) PRODUCT PROFILES — the products your company sells

Your job: for each product in B, assess whether the company signals in A indicate a genuine fit. 
Only recommend a product if you can cite specific company signals that match its ideal_customer_profile or pain_signals.
If a product is not a fit, do not include it.
Be specific: name the product, name the gap it solves, cite the signal.
```

This prevents the LLM from producing a generic "they need a platform" answer and forces it to match specific product capabilities against specific company signals with evidence.

### Maintaining Product Profiles

Product profiles are static config — the team edits them when products change. They are NOT regenerated by the pipeline. Treat them like code: version control them, review changes before committing. If a product's ICP or differentiators change, update the JSON and re-run the pipeline on any companies where a new assessment is needed.

## Schema Design — `schema.py`

This file is the contract between all parts of the system. Define it completely before writing any other file.

### Section 1 — Input Models

```
ScrapedFile
  url: str                          # original source URL this file was scraped from
  file_path: str                    # path on disk
  scraped_at: str                   # ISO date string
  raw_text: str                     # full text content after HTML stripping
```

```
PipelineInput
  company_id: str                   # slug like "sona-comstar" or "oasis-fertility"
  company_domain: str               # e.g. "sonacomstar.com" — used for PRIMARY classification
  files: list[ScrapedFile]
```

### Section 2 — Intermediate Models

```
ChunkWithMeta
  text: str
  source_url: str
  source_type: Literal["PRIMARY", "SECONDARY"]   # set at ingest, before LLM
  scraped_at: str
  chunk_index: int
```

```
SourcedClaim
  value: Optional[str]
  source_url: Optional[str]
  source_type: Literal["PRIMARY", "SECONDARY", "INFERRED"]
  confidence: Literal["HIGH", "MEDIUM", "LOW"]
  reasoning: Optional[str]          # only populate when INFERRED
```

`SourcedClaim` is the atomic unit. Every extractable field in the output is a `SourcedClaim`, not a raw string. This is what enables the source attribution the SRBI reports show.

### Section 3 — Extraction Cluster Models (used internally by Instructor)

These are the models Instructor validates against. One per extraction cluster. They are simpler than the final output — they just need to be extractable from a few retrieved chunks.

```
OverviewCluster
  company_name: Optional[str]
  founded_year: Optional[str]
  headquarters: Optional[str]
  industry: Optional[str]
  business_description: Optional[str]
  source_urls: list[str]

FundingCluster
  total_raised: Optional[str]
  last_round_amount: Optional[str]
  last_round_date: Optional[str]
  lead_investors: list[str]
  pe_backed: Optional[bool]
  source_urls: list[str]

ScaleCluster
  employees: Optional[str]
  revenue: Optional[str]
  revenue_cagr: Optional[str]
  facilities: Optional[str]
  geographies: list[str]
  source_urls: list[str]

CapacityGapCluster
  gaps: list[GapItem]               # list of individual gap observations
  source_urls: list[str]

GapItem
  description: str
  evidence_quote: Optional[str]
  source_type: Literal["PRIMARY", "SECONDARY", "INFERRED"]

PainPointCluster
  pain_points: list[PainItem]
  source_urls: list[str]

PainItem
  description: str
  evidence_quote: Optional[str]
  source_type: Literal["PRIMARY", "SECONDARY", "INFERRED"]

TriggerCluster
  triggers: list[TriggerItem]
  source_urls: list[str]

TriggerItem
  date: Optional[str]
  event: str
  significance: Optional[str]
  source_type: Literal["PRIMARY", "SECONDARY", "INFERRED"]

OfferingFitCluster
  matched_products: list[ProductMatch]  # one entry per product that fits
  why_now: Optional[str]
  recommended_entry_points: list[str]
  source_urls: list[str]

ProductMatch
  product_id: str                       # matches product_id in the product profile JSON
  product_name: str
  fit_rationale: str                    # specific company signal that justifies this product
  evidence_quote: Optional[str]
  signal_source_type: Literal["PRIMARY", "SECONDARY", "INFERRED"]

ConflictReport
  field_name: str
  values_found: list[str]
  sources: list[str]
  resolution: Literal["HIGHEST_CONFIDENCE_SOURCE", "MOST_RECENT", "UNRESOLVED"]
```

### Section 4 — Final Output Model

```
CompanyReport
  company_id: str
  generated_at: str                 # ISO timestamp
  
  overview: ReportSection[OverviewCluster]
  funding: ReportSection[FundingCluster]
  scale: ReportSection[ScaleCluster]
  capacity_gaps: ReportSection[CapacityGapCluster]
  pain_points: ReportSection[PainPointCluster]
  recent_triggers: ReportSection[TriggerCluster]
  offering_fit: ReportSection[OfferingFitCluster]
  
  conflicts: list[ConflictReport]
  overall_confidence: Literal["HIGH", "MEDIUM", "LOW"]
  sources_used: list[SourceSummary]  # deduplicated list of all sources
  inferred_claims_count: int
  null_fields_count: int

ReportSection[T]
  data: T
  section_confidence: Literal["HIGH", "MEDIUM", "LOW"]
  primary_sources_count: int
  secondary_sources_count: int
  inferred_count: int
```

Use a Generic TypeVar for `ReportSection[T]` so the backend can see exactly what type of data each section holds while still sharing the metadata wrapper.

## Pipeline Design — `pipeline.py`

Use `langgraph.graph.StateGraph`. Define a `PipelineState` TypedDict at the top of this file.

### PipelineState Fields

```python
class PipelineState(TypedDict):
    input: PipelineInput
    chunks: list[ChunkWithMeta]
    faiss_index: Any                    # faiss.Index object
    faiss_metadata: list[dict]          # parallel list to faiss index positions
    extraction_results: dict            # keyed by cluster name, value is the cluster object
    conflicts: list[ConflictReport]
    final_report: Optional[CompanyReport]
    errors: list[str]
    retry_count: int
```

### Node Definitions

Define all nodes as plain functions that take `PipelineState` and return a partial dict to merge into state. LangGraph merges partial updates automatically.

**Node 1 — `node_ingest`**
Read all files from the input. Strip HTML if the content is HTML (use `beautifulsoup4`). Validate each file has a URL. Store raw text per file with its metadata into state as the starting chunks (pre-chunking, just one "chunk" per file). If a file is unreadable or empty, log to `state["errors"]` and skip — do not crash.

**Node 2 — `node_classify`**
For each document in the pre-chunk list, classify `source_type` as PRIMARY or SECONDARY using the rule-based classifier from `retrieval.py`. This happens BEFORE chunking so classification metadata propagates to every child chunk. The rule is simple: if the company's own domain is in the URL → PRIMARY. If it matches a known secondary domain list (TechCrunch, YourStory, LinkedIn, Crunchbase, Tracxn, Moneycontrol, Economic Times, BSE, NSE, PitchBook, etc.) → SECONDARY. Default unknown sources → SECONDARY. Never assume PRIMARY for unknown sources.

**Node 3 — `node_chunk`**
Split each document into ~500 token overlapping chunks (50 token overlap). Use `langchain_text_splitters.RecursiveCharacterTextSplitter` with `chunk_size=500, chunk_overlap=50`. Every child chunk inherits the parent document's `source_url`, `source_type`, and `scraped_at`. Store as `list[ChunkWithMeta]` in state.

**Node 4 — `node_embed`**
Batch embed all chunks using `text-embedding-3-small`. Use batches of 100 to avoid rate limits. Build the FAISS index: `faiss.IndexFlatL2(1536)`. Store the index object and a parallel metadata list (same ordering as FAISS index positions) in state. Also persist both to disk at `./indices/{company_id}.faiss` and `./indices/{company_id}_meta.json` so the pipeline can be resumed without re-embedding.

**Node 5 — `node_extract`**
This is the core node. Run all six extraction clusters sequentially. For each cluster:
1. Write a targeted retrieval query (e.g., "company funding raised investors round")
2. Retrieve top 8 chunks from FAISS using `retrieval.py`
3. Format the chunks as context with their source_url and source_type attached
4. Call the appropriate Instructor extractor from `extraction.py`
5. Store the result in `state["extraction_results"][cluster_name]`

Do not parallelize these — sequential is safer for debugging and rate limit management. The six clusters are: overview, funding, scale, capacity_gaps, pain_points, triggers, offering_fit.

For the offering_fit cluster specifically, the retrieval query must pull from ALL previously extracted signals — construct the query from the capacity gaps and pain points already extracted in this same node execution. This is why offering_fit is extracted last.

**Node 6 — `node_infer`**
Run a separate Instructor call specifically for INFERRED claims. This node looks at what was NOT found (null fields in the extraction results) and asks the LLM: given the signals present, what can be logically deduced that no single source explicitly stated? The model MUST set `source_type = "INFERRED"` and populate the `reasoning` field for every claim it generates here. This maps to the INFERRED rows in the SRBI report sources section.

The prompt for this node is strict: "You are generating inferences only. Do not repeat claims already extracted. Only generate a claim if you can cite at least two independent signals that together imply it. If you cannot, output an empty list."

**Node 7 — `node_conflicts`**
Scan all extraction results for fields where multiple sources gave different values for the same thing. Examples: employee count from LinkedIn vs company website, revenue from two different filings. For each conflict detected, create a `ConflictReport`. Attempt automatic resolution: prefer PRIMARY over SECONDARY, prefer more recent over older. If resolution is ambiguous, set `resolution = "UNRESOLVED"` and flag for human review. Store `list[ConflictReport]` in state.

**Node 8 — `node_assemble`**
Assemble all extraction results into the final `CompanyReport`. Compute per-section confidence: HIGH if 2+ PRIMARY sources and no UNRESOLVED conflicts, MEDIUM if mostly SECONDARY, LOW if majority of fields are null or INFERRED. Compute overall confidence as the minimum across sections. Write the final JSON to `./outputs/{company_id}.json`. Store in state.

### Graph Wiring

```
START → node_ingest → node_classify → node_chunk → node_embed 
      → node_extract → node_infer → node_conflicts → node_assemble → END
```

Add a conditional edge after `node_extract`: if `errors` list has grown beyond 3 entries or all extraction results are null, route to a `node_fail` that writes a partial JSON with the error log rather than crashing silently.

## Retrieval Design — `retrieval.py`

### Section 1 — Source Classifier

Implement `classify_source(url: str, company_domain: str) -> Literal["PRIMARY", "SECONDARY"]`.

Maintain these lists as module-level constants:

`ALWAYS_PRIMARY_PATTERNS`: company's own domain (passed in), `bse.com`, `nseindia.com`, `mca.gov.in`, `sebi.gov.in`, `linkedin.com/company` (official company page only, not profiles), regulatory filing domains.

`ALWAYS_SECONDARY_PATTERNS`: `techcrunch.com`, `yourstory.com`, `inc42.com`, `theken.in`, `economictimes.com`, `moneycontrol.com`, `crunchbase.com`, `tracxn.com`, `pitchbook.com`, `bloomberg.com`, `reuters.com`, `livemint.com`, `businessstandard.com`, `entrackr.com`, `vccircle.com`, `dealstreetasia.com`.

Default: if URL matches neither list → `"SECONDARY"`. Never default to PRIMARY for unknown domains.

### Section 2 — Chunking Wrapper

Implement `chunk_document(text: str, metadata: dict, chunk_size: int = 500, chunk_overlap: int = 50) -> list[ChunkWithMeta]`. Wraps `RecursiveCharacterTextSplitter`. Every chunk inherits all metadata from the parent document dict.

### Section 3 — Embedding

Implement `embed_texts(texts: list[str]) -> list[list[float]]`. Uses OpenAI `text-embedding-3-small`. Batches in groups of 100. Includes exponential backoff retry on rate limit errors using `tenacity`.

### Section 4 — Index Management

Implement:
- `build_index(chunks: list[ChunkWithMeta]) -> tuple[faiss.Index, list[dict]]`
- `save_index(index, metadata, company_id: str)` — saves to `./indices/`
- `load_index(company_id: str) -> tuple[faiss.Index, list[dict]]` — loads from disk if exists

### Section 5 — Retrieval

Implement `retrieve(query: str, index, metadata_store: list[dict], top_k: int = 8, source_type_filter: Optional[str] = None) -> list[ChunkWithMeta]`.

Use the overfetch-then-filter pattern: fetch `top_k * 3` results from FAISS, then filter by `source_type_filter` if provided, then return top `top_k`. Format returned chunks as context strings that include the source_url and source_type inline so the LLM sees them.

## Extraction Design — `extraction.py`

### Section 1 — Instructor Client Setup

```python
import anthropic
import instructor

raw_client = anthropic.Anthropic()
client = instructor.from_anthropic(raw_client)
```

This is the singleton client used by all extractors. Do not instantiate it per-call.

### Section 2 — Base Extraction Function

Implement `extract_cluster(context_chunks: list[ChunkWithMeta], response_model: Type[T], system_prompt: str, user_prompt: str, max_retries: int = 3) -> T`.

This function:
1. Formats chunks into a context block with source URLs and types included inline
2. Calls `client.chat.completions.create` with `response_model=response_model, max_retries=max_retries`
3. Returns the validated Pydantic object
4. On total failure after retries, returns a default empty instance of the model (all fields None or empty lists)

Never raise — return an empty model on failure and log the error. The pipeline must complete even with partial failures.

### Section 3 — System Prompt Template

All extractors share this base system prompt, with the cluster-specific instruction appended:

```
You are a structured intelligence extractor for company research reports.

STRICT RULES:
1. You may ONLY use information present in the context chunks below.
2. If a piece of information is not found in the context, return null for that field. Do not guess, infer, or use your training knowledge.
3. For every field you populate, the source_url must come from one of the chunks below.
4. If you are combining two or more signals to conclude something no single source explicitly states, you MUST set source_type to INFERRED and explain your reasoning in the reasoning field.
5. Mark confidence HIGH if 2 or more independent sources confirm it. MEDIUM if only one source. LOW if the evidence is ambiguous.
6. Return empty lists, not null, for list fields when no evidence is found.

Context chunks (each includes its source URL and source tier):
{context_block}
```

### Section 4 — Per-Cluster User Prompts

Write a specific user prompt for each cluster. These prompts tell the model exactly what to look for and what field maps to what concept. Do NOT make them generic. Examples:

**Overview prompt**: "Extract the company's basic profile. Business description should be 2-3 sentences summarizing what the company does, who it serves, and its strategic position. Do not copy marketing language — rephrase in neutral analytical language."

**Funding prompt**: "Extract all investment and funding information. For PE-backed companies, note the PE firm name, investment amount, and any stated mandate or strategic direction the PE board has set. This is important context."

**Capacity Gap prompt**: "Identify specific operational gaps — situations where the company has scale or complexity that its current internal tools, systems, or IT team cannot serve. Look for statements about IT backlogs, manual workarounds, Excel-based processes, disconnected systems, or statements from leadership about digital transformation challenges. Each gap must have evidence."

**Pain Points prompt**: "Extract documented pain points — operational problems causing cost, delay, or visibility failures. Look for cost blowouts, fragmentation across locations, data silos, compliance issues, management visibility gaps. Distinguish between pain points that are publicly documented (PRIMARY) vs inferred from structural signals."

**Triggers prompt**: "Extract recent trigger events — specific dated events that create urgency or change the company's situation. Examples: new JV, new facility, PE investment, leadership hire, new product line, regulatory filing, public statement. Include the date if available and why this event matters."

**Offering Fit prompt**: "Based on the capacity gaps and pain points already identified in this session, describe what a software/platform vendor could supply to solve them. Be specific about what product categories would address each gap. This section synthesizes the evidence — mark all claims INFERRED unless a source explicitly mentions buying intent."

### Section 5 — Individual Extractor Functions

Implement one function per cluster: `extract_overview()`, `extract_funding()`, `extract_scale()`, `extract_capacity_gaps()`, `extract_pain_points()`, `extract_triggers()`, `extract_offering_fit()`. Each function takes `list[ChunkWithMeta]` and returns the typed cluster model. Each calls `extract_cluster()` with its specific `response_model` and prompts.

## CLI Design — `run.py`

Use `typer` for the CLI (cleaner than argparse, produces better help text).

```bash
python run.py --company-id sona-comstar \
              --input-dir ./data/sona-comstar/ \
              --company-domain sonacomstar.com \
              --output-dir ./outputs/
```

The CLI:
1. Reads all `.txt`, `.html`, `.json` files from `--input-dir`
2. For `.json` files, expects the format `{"url": "...", "text": "...", "scraped_at": "..."}` — this is the standard handoff format from the scraping team
3. For `.txt` and `.html` files, requires a companion `.meta.json` file in the same directory with `{"url": "...", "scraped_at": "..."}` — document this clearly in README
4. Builds `PipelineInput` and runs the LangGraph graph
5. Prints progress to stdout at each node transition
6. Writes final JSON to `--output-dir/{company_id}.json`
7. Prints a summary: how many sources used, how many fields populated, how many INFERRED, overall confidence, any UNRESOLVED conflicts

## Input Format — What the Scraping Team Must Provide

The preferred format is one JSON file per scraped source URL. Codex must document this format clearly in README.

**Option A — Preferred (JSON per source)**
```json
{
  "url": "https://techcrunch.com/2024/03/example-article",
  "scraped_at": "2026-03-27",
  "text": "Full plain text content of the page after HTML stripping..."
}
```

One `.json` file per source, named anything (e.g., `source_001.json`), placed in a flat folder per company.

**Option B — HTML files with companion metadata**
Place `page.html` and `page.meta.json` in the same directory. The `.meta.json` must contain `{"url": "...", "scraped_at": "..."}`. The pipeline will strip the HTML itself using BeautifulSoup.

**Option C — Plain text with companion metadata**
Place `page.txt` and `page.meta.json` in the same directory. Same meta format.

The pipeline reads all three formats from the input directory automatically. Mixed formats are supported.

Minimum viable input per file: the source URL. Everything else can be defaulted. Without the URL, source classification is impossible and the file is skipped with a warning.

## Hallucination Prevention — Critical Behaviors Codex Must Implement

These behaviors are non-negotiable and must be validated with tests.

**1. Null-on-absence**: Every `Optional` field in an extraction cluster MUST be nullable. The Instructor prompt must say "if not found, return null" explicitly. Test this by running extraction on a context where the field is provably absent — the output must be null, not a guessed value.

**2. Source URL tracing**: Every extracted cluster must include a `source_urls` field. Instructor must be prompted to only include URLs that appeared in the context chunks. Post-extraction, validate that every URL in `source_urls` actually exists in the chunks that were retrieved. If a URL appears that was not in the context, it is a hallucination — log it and nullify the claim.

**3. INFERRED tagging**: In `node_infer`, the system prompt must prohibit the model from setting `source_type = "INFERRED"` for claims that were already extracted from direct sources. INFERRED is only for novel deductions that combine signals.

**4. No training knowledge**: The prompt must say "Do not use information from your training data. If the context chunks do not contain this information, return null." This is not optional phrasing.

## Error Handling Policy

- **Per-file errors** (unreadable, no URL, empty): log to `state["errors"]`, skip the file, continue. Never crash.
- **Embedding errors** (API rate limit): retry with exponential backoff via `tenacity`. Max 5 retries with 60 second cap.
- **Extraction errors** (Instructor validation failure after max_retries): return empty cluster model (all null/empty list), log the error, continue to next cluster.
- **Total extraction failure** (all clusters returned empty): route to `node_fail` via conditional edge, write a partial JSON with just the error log and company metadata, exit cleanly.
- **Index load failure** (corrupted FAISS file): rebuild from scratch, do not crash.

The pipeline must always produce an output file, even if that file is a partial result with errors documented. The backend team needs to know the pipeline ran even if data quality is low.

## What Not to Build

- No FastAPI server. This is a CLI pipeline. If the team wants an API wrapper later, that is a separate project.
- No database. FAISS indices on disk and JSON output files are the only persistence.
- No LangSmith or observability integration. Log to stdout with Python's `logging` module only.
- No async/await. The pipeline runs synchronously. Async adds complexity that is not needed for a single-company-at-a-time pipeline.
- No Docker. The README installs with pip. Docker is the team's choice to add later.
- No web scraping. This pipeline only processes already-scraped content. Scraping is a separate concern owned by the scraping team.
- No LangChain core. Only `langchain-text-splitters` is imported from the LangChain ecosystem.
- No separate config file or settings module. Use `pydantic-settings` with a `Settings` class defined at the top of `pipeline.py`.
